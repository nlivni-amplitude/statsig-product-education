# Feature Flags / Feature Gates

Statsig's flagging surface. Covers targeting rules, overrides, user identifiers, environments, scheduled rollouts, diagnostics, and the promotion of gates into A/B tests.

## Learning Objectives

### LO1. Construct a targeting rule using attributes and conditions.

**Prompt:** *You want to roll out a new checkout flow to Pro-plan users in Germany only, but exclude anyone in your internal `qa_users` cohort. Describe the targeting rule you'd build.*

**Canonical answer:** A rule with three conditions, all required (AND):
- `plan == "pro"`
- `country == "DE"`
- `qa_users` cohort: NOT IN

You'd also set the **pass percentage** for the rule (e.g., 100% if you want all matching users to get the flow, or a smaller % for a gradual rollout). Anyone who doesn't match all three conditions falls through to the next rule or to the default value.

---

### LO2. Use overrides correctly vs. when to fix the rule.

**Prompt:** *A product manager asks you to "just override this user to see the new flow." When is that the right call vs. when should you fix the targeting rule?*

**Canonical answer:**
- **Override is right** when it's a one-off — testing as a specific user, demo'ing to a customer, debugging why someone's not getting in. Manual user-level opt-in/out.
- **Fix the rule** when the symptom reveals a flaw in the logic. If the PM is overriding 10 users because the rule misses them, the rule is wrong — overrides are masking the bug.
- **Test Gate tool** lets you simulate the rule against a hypothetical user without touching real data.

Overrides are short-lived patches. Long-lived rule errors get worse the more overrides pile up.

---

### LO3. Choose the right user identifier.

**Prompt:** *A user lands on your homepage and signs up halfway through a funnel. You want to A/B test a CTA change. Which identifier do you use for the test unit, and why?*

**Canonical answer:** **Stable ID.** Reasoning: pre-auth visitors don't have a User ID yet, but you still need a consistent unit of analysis so the same visitor sees the same variant across the funnel. Stable ID is auto-generated by the Statsig SDK on first visit and persists (cookie/localStorage on web, device storage on mobile).

If you used User ID, anonymous visitors would get re-bucketed every page load, breaking the experiment.

Once the user signs in, you'd usually switch to User ID for post-auth experiments. Some teams alias Stable ID → User ID to stitch the journey.

---

### LO4. Gate client-side vs. server-side.

**Prompt:** *Where does the gating decision live for this feature: a new pricing page UI, OR a new recommendation algorithm in the backend? Justify each.*

**Canonical answer:**
- **New pricing page UI** → **client-side gate.** Context is in the browser (which variant the user sees, the layout). Latency matters; you want the SDK to evaluate locally.
- **New recommendation algorithm** → **server-side gate.** Context is in the backend (the model, the data). The client doesn't need to know which algorithm ran; it just gets the results. Server-side keeps the implementation hidden and avoids leaking model code to the client.

**General rule:** put the gate inside the service whose behavior is being changed.

---

### LO5. Set up a scheduled rollout for risk.

**Prompt:** *You're rolling out a payments change that could break revenue if it has a bug. How do you use Statsig's rollout features to de-risk?*

**Canonical answer:** Use a **scheduled rollout** with progressive percentages. Example:
- Day 1: 1% — let it bake; monitor exposure stream, error rates, and the primary metric (e.g., payment success rate)
- Day 2: 5% — if metrics hold, expand
- Day 3-4: 25%, then 50%
- Day 5: 100%

At each step the platform compares the gated cohort to control. If a guardrail metric (errors, latency, conversion) regresses, halt the rollout — even revert to 0% with a single config change, no redeploy needed.

This is the killer use case for flags vs. raw deploys: rollback is a UI toggle, not a hotfix.

---

### LO6. Explain how a gate promotes to an A/B test.

**Prompt:** *Statsig says feature gates "automatically become A/B tests." What does that mean concretely, and when is the promotion appropriate?*

**Canonical answer:** When you roll a gate out to a percentage of users (e.g., 50%), Statsig logs which users were in the **pass** group vs. the **fail** group. With those two cohorts and your primary metrics, the platform can compute a scorecard — same statistical machinery as an explicit A/B test — comparing behavior between "got the feature" and "didn't."

**Appropriate when:** the rollout is meant to measure impact (you genuinely care if the feature helped). Use the gate-as-test pattern for opportunistic measurement.

**Not appropriate when:** the rollout is a pure delivery mechanism (e.g., dark-launching infrastructure). There's nothing to measure, and treating it as an experiment adds noise.

---

### LO7. Distinguish a Feature Flag from a Dynamic Config.

**Prompt:** *What's the difference between a Feature Flag and a Dynamic Config? When do you reach for each?*

**Canonical answer:**
- **Feature Flag** — boolean (or simple variant): *whether* this user gets the feature. Use for on/off and A/B/C variants.
- **Dynamic Config** — JSON payload: *how* the feature behaves. Use for tunable parameters (button text, threshold values, weights, configurable copy) where you want to change without redeploying.

**Reach for a flag** when the question is binary: do they see this or not?
**Reach for a config** when the question is parameterized: how big, what text, which value?

Often used together: a flag gates the new feature, a config tunes its behavior.

---

### LO8. Contrast with Amplitude Experiment flags.

**Prompt:** *How are Statsig Feature Gates different from Amplitude Experiment flags? Where is the difference most visible to a developer migrating between them?*

**Canonical answer:** Both products do the same fundamental job: evaluate a user against targeting rules and return a variant.

**Differences:**
- **Coupling to experiments** — Statsig treats a gate as a potential A/B test by default (auto-promotion). Amplitude Experiment historically separates the flag from the experiment more cleanly; an experiment uses a flag but the flag is its own object.
- **Dynamic Config first-class** — Statsig has Dynamic Configs as a top-level concept; Amplitude Experiment supports payloads but they're more of an attribute of the experiment than a standalone object.
- **API shape** — Statsig's `checkGate("name", user)` returns a bool. Amplitude Experiment's `fetch().get("flag-key")` returns a variant object. Slightly different mental models but functionally equivalent for the common case.
