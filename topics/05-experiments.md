# Experiments

Designing, running, validating, and interpreting A/B and multivariate tests in Statsig. Covers hypothesis formulation, exposure logging, diagnostics, scorecard reading, interaction effects, and holdouts.

## Learning Objectives

### LO1. Frame a hypothesis with action + predicted outcome + rationale.

**Prompt:** *Write a well-formed hypothesis for an experiment you'd run. Use the three-part structure (Action, Predicted Outcome, Rationale).*

**Canonical answer:** A good hypothesis has all three parts. Example:
- **Action:** *If we reduce the sign-up form from 8 fields to 4*
- **Predicted Outcome:** *then sign-up completion rate will increase by ≥10%*
- **Rationale:** *because users spend less time and friction, leading to higher completion*

**Why all three matter:** Action without outcome is a feature spec, not a hypothesis. Outcome without rationale leaves you unable to explain a null result (you don't know why you thought it would work). Rationale-without-prediction is a wish.

A well-framed hypothesis aligns stakeholders on what success looks like *before* the data comes in, preventing post-hoc rationalization.

---

### LO2. Choose the right experiment unit.

**Prompt:** *You're running an experiment on a pre-authentication funnel (visitors who haven't signed up yet). What's your experiment unit, and why?*

**Canonical answer:** **Stable ID** (auto-generated by the Statsig SDK), not User ID.

**Why:** anonymous visitors don't have a User ID yet. If you used User ID as the unit, you'd be bucketing on `null` (or whatever default), which means every visitor either falls into a single arm or gets re-bucketed each visit. Stable ID persists across visits via cookie/localStorage, giving you a stable unit of analysis.

After sign-up, you typically transition to User ID for post-auth experiments. Some teams alias Stable ID → User ID at signup to maintain continuity for users who were already mid-experiment.

---

### LO3. Differentiate primary, secondary, and guardrail metrics.

**Prompt:** *You're running an experiment to test a new onboarding flow. Choose your primary, secondary, and guardrail metrics. Explain why each is in that category.*

**Canonical answer (example):**
- **Primary:** onboarding completion rate. This is the metric the experiment is *designed to move*. Decisions are made against it.
- **Secondary:** time-to-first-key-action (e.g., time to first dashboard creation). Diagnostic — explains *how* the primary metric moved.
- **Guardrail:** D1 retention. You don't expect onboarding to move retention much, but if a "faster" onboarding hurts retention, you want to catch it before declaring victory.

**Why category matters:** the platform applies statistical corrections based on how many metrics you treat as primary. Declaring everything primary inflates false-positive risk (multiple comparisons). Declaring nothing primary leaves you cherry-picking after the fact.

### LO4. Validate the experiment before declaring it live.

**Prompt:** *You just launched an experiment. Before you trust the results, what do you check using Diagnostics?*

**Canonical answer:** Use the **Diagnostics** tab to confirm:
- **Exposure stream is firing** — users are actually being assigned. Zero exposures = SDK isn't reaching the experiment.
- **No missing identifiers** — every exposure has a unit ID. Missing IDs = your user object isn't passing what the rule needs.
- **Sample Ratio** matches the allocation — if you set 50/50, exposures should be ~50/50 per arm. Statsig flags **SRM** (Sample Ratio Mismatch) automatically.
- **Exposure-to-event continuity** — users who get exposed should be able to fire the primary-metric event. If exposed users never trigger the metric, something's broken in your funnel.

Skipping diagnostics is how teams ship "winning experiments" that turn out to have been broken from day one.

### LO5. Read a scorecard and decide.

**Prompt:** *A scorecard shows a +3% lift on the primary metric with a 95% confidence interval of [+0.5%, +5.5%] and p=0.02. What's your decision, and what would change it?*

**Canonical answer:** **Ship the variant.** The CI excludes zero (so the effect is statistically distinguishable from no change), the p-value is below the typical 0.05 threshold, and the lower bound is positive (even the pessimistic interpretation is a win).

**What would change my call:**
- If the lower bound were just above zero (e.g., [+0.1%, +5.5%]), I'd want a longer run or a confirmation experiment — the variant *might* be break-even.
- If guardrails regressed (retention dropped, errors spiked), the +3% on primary doesn't matter; the change is net-negative.
- If the absolute lift is +3% but the cost-to-ship is high (a major engineering rewrite), the lift has to clear the bar of "worth that cost."
- If we only ran for 2 days and the metric is week-long retention, the data isn't mature; wait.

### LO6. Reason about interaction effects.

**Prompt:** *Two experiments are running simultaneously on overlapping users. Experiment A tests a new homepage; Experiment B tests a new checkout button. What's the risk, and how do you mitigate it?*

**Canonical answer:** **Risk:** if a user is in Experiment A's treatment AND Experiment B's treatment, their behavior reflects the combination — not either change in isolation. If A and B interact (e.g., a homepage change leads more people to checkout, where they encounter the new button), neither scorecard tells you the isolated effect.

**Mitigations:**
- **Layers / mutually exclusive groups** — Statsig's Layers feature buckets users into a single experiment per layer, so a user in A can't also be in B if they're in the same layer.
- **Holdout analysis** — keep a percentage out of both experiments to serve as a clean baseline.
- **Accept the noise** if the experiments are on truly separate surfaces with minimal user overlap, and document the limitation.

---

### LO7. Use Holdouts and recognize meta-insights.

**Prompt:** *What's a Holdout in Statsig, and what kind of insight does it give you that a single experiment doesn't?*

**Canonical answer:** A **Holdout** is a population you keep *out* of one or more experiments (kept on the control variant) for an extended period. It serves as a long-running baseline.

**Meta-insight** comes from comparing the holdout cohort to everyone-else over time — across many experiments. Individual experiments tell you "did this specific change move this specific metric." A holdout tells you "did our entire experimentation program move the bottom-line metric." That's the question execs ask: "are we actually improving anything net-net, or are we just declaring lots of small wins that average out?"

If your holdout cohort has nearly identical bottom-line metrics to the rest of your users after 6 months of experiments, you've been chasing noise.

---

### LO8. Recognize SRM and what it implies.

**Prompt:** *You see a Sample Ratio Mismatch warning. What does it mean, and what do you do?*

**Canonical answer:** **Sample Ratio Mismatch (SRM)** = the actual split of users between arms doesn't match the configured split (e.g., you configured 50/50 but you're seeing 47/53 with p<0.001 of being just chance).

**What it means:** something is systematically biasing assignment. The experiment's validity is compromised — you can't trust the scorecard.

**Common causes:**
- **Bot traffic** disproportionately in one arm
- **Bucketing bug** in the SDK initialization (e.g., user ID isn't stable across page loads, so users get re-bucketed each visit)
- **Logging bug** — exposures are dropped from one arm
- **Pre-experiment filtering** that's not symmetric across arms

**What to do:** stop the experiment, find the cause, fix it, restart cleanly. Don't ship a result with unresolved SRM.

### LO9. Contrast Statsig experiments with Amplitude Experiment.

**Prompt:** *How do Statsig experiments compare to Amplitude Experiment? Where do they diverge?*

**Canonical answer:** Both run A/B and multivariate tests with traffic allocation, targeting, primary/secondary metrics, scorecards. Both support warehouse-native modes. Differences:
- **Default metric normalization** — Statsig per-user normalizes by default; Amplitude has options.
- **Layers** — both products handle the mutually-exclusive-experiments problem; the implementations differ (Statsig Layers vs. Amplitude shared layers).
- **CUPED variance reduction** — both have it; Statsig markets it more prominently as a first-class feature.
- **Multi-arm bandits** — Statsig has them as a built-in option; Amplitude relies on integrations or doesn't have a direct equivalent.
- **Scorecard visual** — different layouts; same underlying statistical concepts (CIs, p-values, lift).
