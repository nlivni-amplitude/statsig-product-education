# SDKs — Setup and Initialization

How Statsig's SDKs work: client vs server split, init, user object, and advanced setup options (bootstrapping, data adapters, forward proxies).

## Learning Objectives

### LO1. Decide whether a use case calls for a client SDK or a server SDK.

**Prompt:** *You're deciding between a client SDK and a server SDK for a given feature. Walk through how you'd choose. Give one example where each is the right call.*

**Canonical answer:**
- **Client SDK** — runs in the user's browser or mobile app. Use when most of the context is in the frontend (e.g., registration UX, A/B-testing a new landing page layout) or when the feature is primarily a UI/presentation concern. Latency-sensitive UI decisions belong here.
- **Server SDK** — runs on your backend. Use when the context lives on the server (auth state, paid tier, complex permission logic) or when the feature is backend behavior (new cache layer, different API path, ML model swap).
- **Rule of thumb:** localize the gating decision inside the service whose behavior is changing. Don't reach across the wire to gate something the backend already knows.

---

### LO2. Define a user object with the attributes needed for targeting.

**Prompt:** *What goes into the SDK user object, and why does the shape of this object matter for downstream targeting and exposure logging?*

**Canonical answer:** The user object is what Statsig evaluates targeting rules against. It typically includes:
- **User ID** (or Stable ID for pre-auth users — auto-generated by the SDK)
- **Standard attributes** — email, country, locale, app version, OS
- **Custom attributes** — anything specific to your product (plan tier, feature cohort, signup source)

It matters because Statsig can only target on what's in the user object. If your targeting rule says "users on Pro plan in Europe," but you don't pass `plan` and `country`, no user will ever match — silently. Also: PII shouldn't go in raw; hash or omit.

---

### LO3. Use Diagnostics to validate assignment before shipping.

**Prompt:** *You initialized the SDK and shipped a feature behind a gate. How do you confirm assignment is actually working before launch? What signal would tell you something's wrong?*

**Canonical answer:** Use the **Diagnostics tab** in the Statsig console. It surfaces a live exposure stream — every gate check that fires gets logged. Confirm:
- The right user attributes are coming through (no missing identifiers)
- Exposure volume matches what you'd expect for the traffic
- No "user evaluated with no matching rule" warnings

**Warning signs:** zero exposures (SDK isn't firing or user object is empty), every user falling into the default group (rules aren't matching), mismatched exposure vs. event volume (your event logging is broken even if gates work).

### LO4. Decide when to use bootstrapping, data adapters, or a forward proxy.

**Prompt:** *Statsig's advanced setup includes bootstrapping, data adapters, and forward proxies. What problem does each solve? When should you reach for them?*

**Canonical answer:**
- **Bootstrapping** — pass pre-computed evaluations from your server to the client at page load. Solves the "first-render flash" problem where the client SDK has to do a network call before it knows the user's variant.
- **Data adapters** — let the SDK cache/fetch config values from your own infrastructure (Redis, your CDN, etc.) instead of Statsig's edge. Solves resilience and latency in regions where Statsig's CDN is slow.
- **Forward proxies** — proxy SDK traffic through your own infrastructure. Solves compliance/privacy concerns where you don't want the SDK calling out to a third-party domain directly.

None are required for a basic setup. Reach for them when you have a specific symptom (flash, latency, compliance).

---

### LO5. Handle PII correctly in the user object.

**Prompt:** *Which attributes should NOT go raw into the user object, and how do you handle them?*

**Canonical answer:** Don't pass raw PII — names, raw email addresses, phone numbers, addresses. If you need to target by something that *is* PII:
- **Hash before passing** (one-way hash so Statsig sees a stable token, not the underlying value)
- **Bucket and pass the bucket** ("US East", not the IP address)
- **Omit if not needed** for targeting — most attributes are optional

Same principle applies to logging: don't put PII in event names or property values.

---

### LO6. Contrast Statsig's "one SDK does everything" pattern with Amplitude's split.

**Prompt:** *How does Statsig's SDK approach differ from Amplitude's? Where does the difference matter?*

**Canonical answer:** Statsig bundles flag evaluation + experiment assignment + exposure logging + analytics events into one SDK with one init. Amplitude historically splits this: Amplitude Analytics SDK for events + Amplitude Experiment SDK for flag/variant evaluation.

**Where it matters:**
- **Setup friction** — Statsig is one package and one init; Amplitude is two (more wiring, but more modular).
- **Cohesion** — Statsig auto-logs an exposure when you check a gate; in Amplitude you have to wire exposure → analytics yourself (or use the SDK integration).
- **For migrators** — a customer moving from Statsig to (Amplitude Analytics + Amplitude Experiment) needs to split what was one call into two.
